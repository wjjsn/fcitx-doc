# 开发者问答

这是一个回答一些问题页面。
有些问题经常被问到，其他的则是一些值得回答/提及的常见陷阱。

## 我不喜欢 Fcitx 的默认 UI，想为自己的引擎拥有一个自己的 UI。

这是第三方专有输入法引擎开发者经常问的问题。从 Fcitx 开发者的角度来看，答案是**不**，不要这样做。这个答案背后的原因是由于 Linux 显示服务器的复杂情况。

- X11，你可以自由移动窗口。
- Wayland + Fcitx 输入法模块，窗口可能由客户端渲染，你基本上无法控制。我们不能让 Fcitx 输入法模块有另一个插件系统，这会带来太多复杂性，并且可能会给应用程序带来太多内存占用。
- Wayland + text-input + zwp-input-method，弹出 surface 需要使用特殊的 wayland 协议， random 进程可能无法访问。只有输入法服务器可能有权限显示这样的输入窗口。

总之，由于 wayland 的限制，如果您选择实现自己的版本，可能无法正确显示输入窗口。

如果您仍然想这样做，您应该意识到的是，您可能不想实现一个新的 Kimpanel 或新的用户界面插件。对于 Kimpanel，它旨在与桌面集成，并且只能有一个 kimpanel 服务。如果您想实现 Kimpanel，桌面本身可能已经有一个 Kimpanel 服务在运行。没有这样的机制来替换另一个 kimpanel 并将 kimpanel dbus 名称返回给原始的。在实现新的 Kimpanel 或新的用户界面时，它必须是通用的并支持所有其他现有输入法。如果您认为这不是您想要的，您最好只为自己实现 UI。这意味着您应该避免使用 Fcitx 中所有内置的 UI 机制，除了客户端预编辑。只需简单地为 InputContext::inputPanel() 在 Fcitx 中不设置任何值并保持为空。对于您的专有用户界面进程，您可能需要使用进程间通信机制（如 DBus）来发送信息。您可以重用像 Kimpanel 这样的 dbus 接口，但请确保只使用您自己的 DBus 名称，不要尝试替换现有的 Kimpanel dbus 服务。

## 有些选项看起来可以从全局配置中重用，我应该使用它吗？

对于输入法引擎，一些全局配置选项如上一页键、下一页键，可能会被使用，但您可能也想使用您自己配置中的那个。对于每个输入法引擎来说，没有这样的默认值是好的。此默认选项旨在供不想提供自己上一页键和下一页键的插件使用。

例如，很多拼音用户通常使用减号和等号键作为翻页，但如果您不使用美国键盘布局，这样的键可能不是一个好选择。在全局配置中，我们只分配了可能适用于每个键盘布局的非常通用的键，例如方向键。

## 我如何测试输入法？

Fcitx 提供了一个小型测试框架，允许您模拟按键事件。请参考 <code>fcitx-utils/testing.h</code> 了解更多详细信息。另请参阅 [testquickphrase.cpp](https://github.com/fcitx/fcitx5/blob/master/test/testquickphrase.cpp) 作为如何模拟按键事件并将其发送到假输入上下文的好例子。

## 我应该使用周围文本功能吗？

有很多应用程序永远不会使用它。例如，基于 xim 的客户端不工作（xterm、wine 等），SDL 应用程序。终端也是一个很好的例子，如果您仔细想想的话。

如果在某些应用程序中它们没有使用系统内置的小部件，它也可能有错误，特别是浏览器。此外，如果网页内容在显示文本方面混乱（如 Google Docs），它也可能无法工作，而且很糟糕，因为我们无法检查应用程序的版本，即使我们在某些情况下成功修复了它们。可以通过检查应用程序名称来进行某些破解，但应用程序的名称在某些情况下可能不可用，这取决于使用什么协议。此外，嵌入的终端窗口（如 konsolepart）是应用程序名称检测变通方法失败的情况。

它可以是一个很好的补充信息来源，但我宁愿不依赖它做所有事情。例如，
- 日语通常只使用它进行"重新转换"（选择一系列平假名字符并用选定的文本重新启动组合）
- 韩语也只将其用于临时汉字模式，但它也提供了一种不使用周围文本的始终开启的汉字模式（类似于 unikey 中的宏模式）。
- 在中文中，Fcitx 在某些情况下仅使用它来检测上下文，而不是修改周围的文本。

总之，使用来自周围的信息比修改周围的文本更可靠，并且当它不能正常工作时，有一个简单的方法来解决它（从用户可能找出另一种输入文本的方式的意义上说）会更好。
